# tenki-mcp — test matrix (all key scenarios)

_Decided 2026-07-21 by a PM council (coverage / agent-journeys / risk-edge / real-client perspectives → synthesized). 34 scenarios. Source of truth for what "tested comprehensively" means; run status tracked in TEST-REPORT.md._

**Coverage status:** the per-domain happy paths are green in `test/coverage.test.mjs` (16/16, all 84 tool domains). The fan-out suites cover client-integration, negative/edge, and the composite journeys below.

## client-integration

### [P0] Handshake + protocol negotiation + tools/list advertises all 84 tools with valid schemas
- **id:** `mcp-handshake-toolslist` · **plane:** control
- **steps:** Spawn `node dist/index.js` via StdioClientTransport with a DUMMY TENKI_API_KEY (offline; registration precedes any network call). Send `initialize` (once with the SDK's current protocolVersion, once pinned to an older supported one), then `initialized`, then `tools/list`. Assert serverInfo=={name:'tenki',version:'1.0.1'}, capabilities.tools.listChanged==true, a compatible protocolVersion echoed for both versions. Assert exactly 84 tools; the name set matches the canonical tenki_* list byte-for-byte (diff for missing/extra); every name matches ^tenki_[a-z0-9_]+$ and is unique; every tool has a non-empty description and inputSchema.type==='object' with all required[] fields present in properties.
- **pass:** Handshake completes <2s offline on a dummy key; exactly 84 uniquely-named tenki_* tools with valid object schemas + descriptions; canonical name set matches (any count/name drift = a parity regression); older protocol version negotiates without disconnect. This is the gate that the server loads at all in Claude Desktop/Cursor/Claude Code.
- **cleanup:** client.close(); assert the stdio child is reaped (no orphan node pid). No API resources created.

### [P0] stdout carries only JSON-RPC frames; close/EOF/SIGTERM shut down cleanly with no orphan
- **id:** `stdout-purity-and-clean-shutdown` · **plane:** control
- **steps:** Launch the server capturing stdout and stderr separately. (a) Idle (EOF) launch: assert 0 bytes on stdout and the 'tenki-mcp running on stdio' banner + all logs on stderr. (b) Do a handshake + one tools/list: assert every stdout byte parses as a newline-delimited JSON-RPC message (no stray writes that would corrupt framing). (c) Teardown paths: client.close() after handshake; separately pipe empty stdin (EOF); separately SIGTERM a running instance — measure exit in each.
- **pass:** stdout is pure JSON-RPC (0 bytes when idle; banner/logs only on stderr) — a single stray stdout write breaks framing for every client. Each teardown exits within ~2s (close ~4ms; stdin-EOF exit 0) with no hang and no orphaned dist/index.js pid.
- **cleanup:** Confirm no lingering node pids for dist/index.js. No API resources created.

### [P0] Credential handling end-to-end: missing key exits 1, fallback starts, bogus key surfaces a clean 401
- **id:** `startup-and-auth-contract` · **plane:** control
- **steps:** (a) Launch with NEITHER TENKI_API_KEY nor TENKI_AUTH_TOKEN: assert exit code 1 and stderr 'tenki-mcp: set TENKI_API_KEY (or TENKI_AUTH_TOKEN)...', with no half-open transport. (b) Launch with only TENKI_AUTH_TOKEN set: assert it starts normally (fallback env path). (c) Launch with a well-formed-but-invalid tk_ key and call tenki_whoami: assert an isError result 'Tenki WhoAmI failed (401 ...)' within ~2s (no hang/crash), token value NOT echoed. (d) Repeat (c) with an ory_st_ junk key (X-Session-Token path) and a no-prefix junk key (Cookie path) to exercise all three authHeaders() branches.
- **pass:** No-key launch fails fast (exit 1, documented stderr line); TENKI_AUTH_TOKEN fallback starts; each bad-key whoami returns a clean attributable auth error through the normal MCP tool-result channel across all three prefix branches — never a crash, hang, unhandled rejection, or leaked token.
- **cleanup:** Kill child processes. No API resources created.

### [P1] Tool-description routing: preview-URL 3-way overlap, revoke-share duplication, run-vs-exec, single-vs-bulk terminate safety
- **id:** `routing-disambiguation-audit` · **plane:** control
- **steps:** Give a model only tools/list + a prompt, and record tool selection across repeated runs (do NOT execute destructive picks against real resources). (1) 'I started a web server on port 3000 in sandbox S — give me a public URL' → which of tenki_expose_port / tenki_create_preview_url / tenki_open_preview (all three describe 'a public/preview URL for a port'). (2) 'Revoke workspace W's access to my image' → tenki_unshare_image vs tenki_revoke_image_share_grant (describe-identical, different backend RPCs). (3) 'Run this Python once' (no sandbox) → must pick tenki_run_code; 'run npm test in my existing sandbox sess_123' → must pick tenki_exec. (4) SAFETY: 'destroy sandbox sess_ABC' → must pick tenki_terminate_sandbox (single), NOT tenki_terminate_sandboxes with a fabricated list. (5) Sanity-check the 5 sub-40-char descriptions (list_sandboxes, resume_sandbox, list_files, delete_snapshot, get_template) still win against siblings.
- **pass:** Selection is consistent and correct for each intent. FLAG the description overlaps as findings: the 3-way preview-URL ambiguity, the unshare/revoke duplication (same words, different RPC), and any terse description that loses to a neighbor — with recommended differentiating language. Destructive-safety gate: single-target intent must never select the IRREVERSIBLE bulk verb without an explicit multi-id list.
- **cleanup:** Routing eval only; do not execute destructive picks. If any tool was executed against a live sandbox, unexpose/delete + terminate it.

### [P1] N concurrent tool calls multiplex correctly over one stdio pipe with no cross-talk or leak
- **id:** `concurrency-multiplexing-no-leak` · **plane:** both
- **steps:** On a single connection, without awaiting serially, fire 5x tenki_run_code {language:'python', code:'print("tag-<i>")'} for i in 0..4 concurrently (Promise.all), plus a couple of fast tenki_whoami. Correlate JSON-RPC ids to responses. Then tenki_list_workspace_sandboxes(include_terminated=false) and diff vs the baseline captured before the burst.
- **pass:** All promises resolve; each response id matches its request; no frame interleaving/corruption; a slow call does not starve fast ones (event-loop non-blocking). All 5 run_code return DISTINCT sessionIds with each stdout matching its own tag and NO bleed-over (validates the per-exec capture-file suffix uniqueness + per-session credCache keying). All 5 sessions self-terminate and the running-sandbox count returns to baseline — concurrency leaks nothing.
- **cleanup:** Terminate any of the 5 sessions still RUNNING; re-list to confirm baseline.

## core

### [P0] WhoAmI resolves an owner + a (workspace, project) pair — the dependency of every defaulted call

- **steps:** Call tenki_whoami. - **pass:** Each language boots→runs→returns the right output; env injected; timeout honored; non-zero exit surfaced with exitCode; EVERY ephemeral session is TERMINATED afterward (no billing leak, even on failure). - **cleanup:** tenki_terminate_sandbox on the created id (also the terminating assertion); idempotent if already gone.

### [P0] exec returns inline stdout/stderr, honors cwd + non-zero exit, and surfaces capture failures
- **id:** `exec-inline-exitcodes-and-capture` · **plane:** both
- **steps:** Create a sandbox. (a) tenki_exec {command:'echo', args:['qa-exec']} → ok, exitCode 0, stdout 'qa-exec'. (b) tenki_exec {command:'sh', args:['-c','echo out; echo err 1>&2; exit 3']} → exitCode 3, ok false, stdout 'out', stderr 'err'. ## ship-code

### [P0] THE thesis: red→green test loop (create → write buggy → exec test fails → fix → exec test passes)
- **id:** `ship-code-red-green-loop` · **plane:** both
- **steps:** tenki_create_sandbox {name:'qa-ship-<runid>', cpu_cores:2, memory_mb:4096, allow_outbound:true, wait_ready:true}. tenki_write_file /home/tenki/calc.py = 'def add(a,b):\n    return a-b\n' (buggy). tenki_write_file /home/tenki/test_calc.py = 'from calc import add\nassert add(2,3)==5, f"got {add(2,3)}"\nprint("PASS")\n'. tenki_exec python3 /home/tenki/test_calc.py cwd=/home/tenki → RED: exitCode!=0, stderr AssertionError. tenki_write_file calc.py = 'def add(a,b):\n    return a+b\n' (fix). tenki_exec again → GREEN: exitCode 0, stdout PASS. Optional ship-ready proof: tenki_git operation='init' then add/commit.
- **pass:** The RED run genuinely exits non-zero and the GREEN run exits zero with PASS — the transition is observable. exitCode rides the control-plane ExecuteCommand response, so the loop is provable even if stdout/stderr capture (data plane) degrades; if capture is down, rely on the exitCode flip and FLAG the missing stderr. This is 'agents that ship code' end to end.
- **cleanup:** tenki_terminate_sandbox in a finally (even if a middle step failed).

TOP 7 RISKS TO WATCH (code-grounded, with file refs):
1. NO fetch timeout anywhere — client.ts control()/data() call fetch() with no AbortSignal. The intermittent 100.x data plane HANGS until OS TCP timeout (minutes), which can exceed a real client's tool-call timeout and hang Claude Desktop. Biggest reliability gap. Targeted by resilience-timeouts; any data call hanging >30s = release blocker. Fix: AbortSignal.timeout on both planes.
2. create_sandbox leak asymmetry — runCode() wraps teardown in try/finally (client.ts:302-318) but tenki_create_sandbox (sandboxes.ts:53-58) has NO try/finally around waitForState; a wait_ready timeout/ERROR orphans a billing VM and may not even return its id. Highest-value bug candidate. Targeted by create-sandbox-wait-timeout-leak; backstopped by orphan-sweep. Fix: wrap create+wait in try/finally.
3. Volume quota (~10) + the documented Jul-17 10-volume leak — every volume scenario create+deletes tightly, pre-sweeps qa-* leftovers, and the final orphan-sweep must confirm baseline. resize_volume was broken in 1.0.0 (sent sizeBytes; API wants newSizeBytes, fixed in 1.0.1, volumes.ts:116) — regression-guarded in volume-crud-resize.
4. delete_image empty/whitespace-version footgun — registry.ts:110 (version.trim()!=='') routes a blank/whitespace version to whole-image DeleteRegistryImage instead of DeleteRegistryImageVersion. Probed on a throwaway ref in registry-version-delete-and-sharing; recommend zod-rejecting empty/whitespace version.
5. Data-plane intermittency contract — exec output + file reads ride the 100.x mesh; execCaptured (client.ts:227-274) degrades a capture-read failure into a captureError field with empty stdout while keeping the control-plane exitCode. Rule baked into every 'both' scenario: empty output + captureError/resolve-error == BLOCKED/infra to FLAG, never a code FAIL, while still asserting the control-plane half.
6. Retry asymmetry — control() retries 429/resource_exhausted/unavailable <=3x with capped exponential backoff (client.ts:98-118, cap 8s, attempt<MAX_RETRIES guards infinite loop); data() has ZERO retry (client.ts:144-166). Flagged in resilience-timeouts.
7.
