# tenki-mcp — release roadmap (v0.2 → v2.0)

_Decided 2026-07-20 by a PM council (4 perspectives: developer-experience, parity, adoption/build-in-public, risk/reliability → synthesized). Source of truth for release grouping; STATE.md tracks live progress._

**Principle:** complete the v0.1 partials first in dependency order (files → sessions → previews), then net-new resources grouped by theme, then distribution at v1.0, then the hard streaming/transport leap at v2.0. Every release is individually shippable and live-verified against `api.tenki.cloud` before it's cut.

## v0.2.0 — Filesystem completion (data plane)
**Tools (7):** `tenki_stat_path` · `tenki_make_dir` · `tenki_remove_path` · `tenki_move_path` · `tenki_write_files` · `tenki_upload_file` · `tenki_download_file`

Files are the surface a coding agent touches on nearly every task, yet v0.1 ships only read/write/list. This completes the highest-frequency, lowest-risk resource and is the safest opener (nothing depends on it). Stat/Mkdir/Remove run over the already-live data plane with zero new transport; tenki_write_files batches WriteFile so an agent scaffolds a whole tree in one round-trip; tenki_move_path resolves to the native data-plane Move if present, else an exec/copy+remove shim (the decompiled descriptor lists only Read/Write/Stat/Mkdir/Remove/List, so Move and batch-write are grounded as verifiable client-side conveniences). Binary transfer via signed artifact URLs (GetArtifactUploadUrl/GetArtifactDownloadUrl) unblocks datasets, wheels, images and build artifacts, and the same artifact path can later retire the sh -c exec-capture hack. Verified by appending a create->mkdir->write_files->stat->remove round-trip PLUS an isolated upload->download->checksum-compare to the cheapest-first live smoke test, so a signed-URL regression can't mask text-I/O health. Ship an npm 'next' pre-release the same day for npx ergonomics; the stable publish stays at 1.0.

## v0.3.0 — Session lifecycle & fleet control
**Tools (6):** `tenki_update_sandbox` · `tenki_extend_sandbox` · `tenki_report_activity` · `tenki_terminate_sandboxes` · `tenki_list_workspace_sandboxes` · `tenki_list_project_sandboxes`

Completes the core session object that every other resource hangs off (snapshots capture one, volumes attach to one, previews expose one), so it lands second for dependency-safety. UpdateSession (rename/labels/limits) and ExtendSession push out max-duration/idle so a long build or test run isn't reaped mid-task; ReportSessionActivity is the keep-alive heartbeat; TerminateSessions(bulk) plus the workspace/project listers give an orchestrator driving many parallel agents fleet visibility and cheap teardown. Crucially, keep-alive/extend must exist BEFORE the long-running snapshot and template-build loops arrive in v0.5-v0.6. All unary control-plane calls over the existing retrying client, so no transport risk. Guardrails: TerminateSessions is explicit-id-only (no terminate-all default), ExtendSession affects billing, and the listers honor pagination to stay off the rate limiter. Verified by create two disposable sandboxes -> extend one -> report activity -> bulk-terminate both -> list to confirm.

## v0.4.0 — Preview URLs — the ship surface
**Tools (6):** `tenki_open_preview` · `tenki_create_preview_url` · `tenki_get_preview_url` · `tenki_list_preview_urls` · `tenki_delete_preview_url` · `tenki_unexpose_port`

The 'ship' in 'agents that ship code' and the roadmap's most shareable moment: an agent writes a web app and hands back a public, clickable URL (tenki_open_preview), the proof that turns 'ran code' into 'shipped an app.' Completes the last v0.1 partial (ports) with UnexposePort and adds the managed named-preview lifecycle (Create/Get/List/Delete). Surfaced this early, the first capability past the core completions, precisely because it is the strongest momentum and differentiation wedge, and it only needs v0.1 port exposure to build on. The remaining preview primitives (TouchPreview / BindPreviewUrl / UnbindPreviewUrl / ResolvePreviewToken) round out the resource for the v1.0 parity audit wherever the CLI exposes them. Verified by expose_port -> create_preview_url -> open -> list -> delete -> unexpose against a live sandbox running a trivial server.

## v0.5.0 — Persistent state — snapshots & volumes
**Tools (20):** `tenki_create_snapshot` · `tenki_restore_snapshot` · `tenki_get_snapshot` · `tenki_list_snapshots` · `tenki_list_session_snapshots` · `tenki_list_dangling_snapshots` · `tenki_list_workspace_snapshots` · `tenki_list_project_snapshots` · `tenki_update_snapshot` · `tenki_delete_snapshot` · `tenki_get_snapshot_download_url` · `tenki_create_volume` · `tenki_attach_volume` · `tenki_detach_volume` · `tenki_resize_volume` · `tenki_get_volume` · `tenki_list_volumes` · `tenki_list_project_volumes` · `tenki_update_volume` · `tenki_delete_volume`

The persistent-state story for the iterative agent loop, grouped as one milestone to keep the roadmap within the release cap; each sub-resource ships whole (never half-done) and is independently verifiable. Snapshots are the highest-leverage new primitive for the thesis, checkpoint a known-good state before a risky refactor, branch experiments, restore in seconds if tests fail, with Restore already having a landing spot (create_sandbox's snapshot_id) and GetSnapshotDownloadURL re-exercising the signed-URL path proven in v0.2. Volumes are the complement: persistent disks an agent attaches at create time to carry a node_modules/pip cache or a dataset across otherwise-ephemeral sandboxes, turning cold installs into warm ones (the best benchmark moment on the roadmap). Mostly low-risk control-plane, guardrailed so Delete is explicit-target-only and a volume attached to a live sandbox can't be deleted. Verified by snapshot -> delete-dangling -> restore-into-new-sandbox, and create -> attach -> write-through-a-sandbox -> detach -> re-attach-elsewhere -> delete.

## v0.6.0 — Custom runtimes — templates & registry
**Tools (21):** `tenki_create_template` · `tenki_get_template` · `tenki_list_templates` · `tenki_list_project_templates` · `tenki_update_template` · `tenki_delete_template` · `tenki_build_template` · `tenki_cancel_template_build` · `tenki_get_template_build` · `tenki_list_active_template_builds` · `tenki_publish_image` · `tenki_get_image` · `tenki_list_images` · `tenki_set_image_visibility` · `tenki_delete_image` · `tenki_delete_image_version` · `tenki_resolve_image_ref` · `tenki_share_image` · `tenki_unshare_image` · `tenki_revoke_image_share_grant` · `tenki_list_image_share_grants`

The bring-your-own-runtime tier: define an environment once and boot into it warm instead of reinstalling a toolchain every run, the largest per-iteration latency win, compounding every downstream execute->test->review loop. Templates introduce the platform's first async job surface (BuildTemplate kicks off, GetTemplateBuild polls, CancelTemplateBuild aborts, ListActiveTemplateBuilds monitors); the polling loop must reuse the existing waitForState/backoff so a slow build neither hammers the rate limiter nor hangs the MCP client, and CRUD-without-build (or vice versa) would be a broken resource, so both halves ship together. The registry is the team-scale companion, publish a versioned image, resolve_image_ref to boot from it, share across a workspace, and its ACL surface (SetVisibility/Share/Unshare/RevokeShareGrant/ListShareGrants) ships private-by-default with no implicit public or shared state so a security-relevant resource can't leak. Both already have landing spots in create_sandbox (registry_ref). Sequenced after storage because custom runtimes presuppose users already run plain sandboxes fluently. Verified by build a trivial template -> poll to completion -> boot a sandbox from it, and publish -> resolve -> set-private -> attempt cross-account resolve to confirm visibility is enforced.

## v0.7.0 — Administration & access — workspace & SSH
**Tools (8):** `tenki_get_workspace_usage` · `tenki_get_workspace_sandbox_settings` · `tenki_update_workspace_sandbox_settings` · `tenki_get_snapshot_retention_settings` · `tenki_update_snapshot_retention_settings` · `tenki_update_ssh_keys` · `tenki_issue_ssh_cert` · `tenki_list_ssh_gateways`

The governance and access tail, two small workspace-scoped surfaces combined and placed last so v1.0 can stay a pure packaging milestone with zero new API surface. Workspace administration (GetWorkspaceSandboxUsage for cost/metering, plus the get/update pairs for sandbox settings and snapshot-retention) is the enterprise-readiness and policy surface operators need; SSH access (UpdateSSHAuthorizedKeys on SandboxService, IssueSandboxSSHCert and ListActiveSSHGateways on the SSHGatewayClientService) is the human escape hatch into a sandbox the agent built, and says 'this is not a toy.' These are the lowest agent-invocation-frequency surfaces, invoked by operators and CI rather than the coding agent in its loop, which is why they earn their slot only as a parity requirement. Landing this drives coverage to 100% of every non-streaming method. Guardrails: Update* are standing-config changes affecting cost and data-lifetime, so verify with read-back (get -> update -> get -> restore); cert material is treated as a secret, surfaced once and never logged. Verified by a usage read, a settings round-trip that restores prior values, and issuing an SSH cert against a live sandbox without printing it.

## v1.0.0 — Full CLI parity + distribution
**Tools (5):** `npm publish — stable, semver-locked (npx -y tenki-mcp)` · `MCP registry listing (Anthropic MCP Registry + Smithery/PulseMCP/Glama)` · `parity-audit CI gate — fails build if any unary SandboxService/DataPlane/SSHGateway method lacks a tool` · `per-tool docs + flagship execute->test->review recipe` · `tool-name/schema freeze + consistent error taxonomy`

A certification and distribution milestone with zero new API tools, because by v0.7 every unary SandboxService + SandboxSessionDataPlaneService + SSHGatewayClientService method is already a tool. 1.0 proves and ships that completeness: a CI parity-audit that enumerates every non-streaming method from the service contract and fails the build if one isn't wired to a tool (turning 'full CLI parity' from a claim into a gate and a marketable 'nothing missing'), the stable semver-locked npm publish so any client config becomes npx -y tenki-mcp, the MCP-registry listing for one-search discoverability, complete per-tool docs, and a tool-name/schema freeze plus consistent error taxonomy so downstream clients can depend on it. Stdio only, HTTP/SSE and streaming are deliberately deferred to the v2 arc so nothing blocks the parity milestone. Shippable as a packaging event once v0.2-v0.7 land; nothing here changes tool logic, so it cannot break existing tools. Verified by the parity audit passing and the cheapest-first live-smoke suite (now wired in as a release gate) going green end to end against api.tenki.cloud.

## v2.0.0 — Streaming, interactive execution & HTTP transport
**Tools (7):** `tenki_stream_exec` · `tenki_shell` · `tenki_dial` · `tenki_tunnel_port` · `Connect/gRPC-streaming client transport` · `HTTP+SSE server transport (alongside stdio)` · `framework adapters (Claude Agent SDK / LangChain / LlamaIndex / Vercel AI SDK)`

The genuine architectural leap, quarantined at the major version because it needs a transport the current fetch/JSON-over-HTTP client fundamentally cannot speak. A Connect/gRPC-streaming client unlocks StreamCommandOutput (tenki_stream_exec, live stdout/stderr that retires the sh -c redirect capture workaround and upgrades exec/run_code output fidelity) and the bidi data-plane primitives: interactive Run (tenki_shell, a PTY an agent drives turn-by-turn), Dial (tenki_dial, raw TCP into the sandbox), and HostPortTunnel (tenki_tunnel_port, bring a sandbox port back to the client); the streaming file variants ReadFileStream/WriteFileStream fold into the same transport. It also ships an HTTP+SSE server transport alongside stdio to unlock hosted, multi-tenant deployment, a one-click remote connector with no local Node install, and per-request auth, plus framework adapters to meet agents where they are already built. Major bump: the new transport, per-request auth, and hosted deploy model change config and deploy contracts. Deferring all of it kept the entire request/response roadmap unblocked by the single hardest infra change; it lands only after the unary surface is stable and frozen at 1.0. Verified on its own streaming harness (long-lived connection, backpressure, auth) before it touches any unary tool, with HTTP/SSE validated against a hosted reference deployment. Closes the entire remaining API surface: every method, plain and streaming, is now exposed.

## Key bets
1. Complete the partials first, in dependency order, before opening any net-new resource. v0.1 leaves three resources half-exposed (filesystem, sessions, ports); finishing them at v0.2->v0.4 gives adopters a gap-free CORE and is the highest-frequency, lowest-risk work. Filesystem goes first (touched every task, zero new transport), then the session object (referenced by every downstream resource, and its keep-alive/extend must exist before the long-running snapshot and build loops arrive).

2. Previews are the momentum wedge; land the hero demo by v0.4. A public URL an agent opens turns 'ran code' into 'shipped an app', the single most screenshot/GIF-able proof of the thesis. The one real sequencing tension across the proposals (previews-early vs sessions-first) is resolved by doing sessions at v0.3 for dependency-safety and previews at v0.4, so the demo is still early without shipping the persistence/runtime layers before the core session object is complete.

3. Group by theme to honor the tight release cap. Snapshots+volumes ship as one 'persistent state' milestone (v0.5) and templates+registry as one 'custom runtimes' milestone (v0.6); one-resource-per-release would push this to ~10 releases. Each sub-resource still ships whole and is independently verifiable against the live API, and the grouping is sequenced by agent-invocation frequency and warm-start payoff (state before runtimes before admin), not by API module boundaries.

4. Publish an npm 'next' pre-release at v0.2, but reserve the stable npm publish + MCP-registry listing for v1.0. npx ergonomics for early adopters shouldn't wait for parity, yet 1.0 stays the stable, semver-frozen, registry-listed event the task defines. Back the parity claim with a CI parity-audit that enumerates every unary SandboxService/DataPlane/SSHGateway method and fails the build if one lacks a tool, making 'full CLI parity' a testable gate rather than a marketing line.

5. Quarantine streaming/interactive + HTTP/SSE transport in v2.0 behind a new Connect/gRPC-streaming transport, and ship the entire request/response surface over stdio first. It is the single hardest piece of engineering (connection lifecycle, backpressure, long-lived auth) and the plain-HTTP client cannot speak it; bundling it earlier would hold the whole roadmap hostage to one infra rewrite. Independently, guardrail every destructive/billing/credential verb (bulk terminate, extend, snapshot/volume/image delete, share grants, SSH cert issuance) as explicit-target-only, private/no-op by default, with read-back on standing-config changes, because the failure mode there is data loss or a leaked image/cert, not a 500.
